1 Overview of the package

This document describes version 0.2d0 of CCDoubles, a C99 language library implementing simple routines for vectors and matrices of double and double complex values. This library is meant to be used as companion for more sophisticated libraries such as CBLAS and LAPACKE.

CCDoubles installs the single header file ccdoubles.h. All the function names in the API are prefixed with ccdoubles_; all the preprocessor macro names are prefixed with CCDOUBLES_; all the type names are prefixed with ccdoubles_ and suffixed with _t.

1.1 Linking code with the CCDoubles library

This package installs a data file for pkg-config; usage examples: 

$ pkg-config ccdoubles --modversion
0.1

$ pkg-config ccdoubles --cflags
-I/usr/local/include

$ pkg-config ccdoubles --libs
-L/usr/local/lib64 -lccdoubles

When searching for the installed library with the GNU Autotools, we can:

  1. Install the file pkg.m4 from /usr/share/aclocal into the source tree of the package, for example under meta/autoconf/.
  2. Include pkg.m4 in the template by adding the following line to acinclude.m4: 
    m4_include(meta/autoconf/pkg.m4)
  3. Add the following macro use to configure.ac: 
    PKG_CHECK_MODULES([CCDOUBLES],[ccdoubles >= 0.1])

    which will set the variables CCDOUBLES_LIBS and CCDOUBLES_CFLAGS.

  4. Make use of CCDOUBLES_LIBS and CCDOUBLES_CFLAGS in the Makefile.

Alternatively we can use the raw GNU Autoconf macros: 

AC_CHECK_LIB([ccdoubles],[ccd_version_string],,
  [AC_MSG_FAILURE([test for CCDoubles library failed])])
AC_CHECK_HEADERS([ccdoubles.h],,
  [AC_MSG_FAILURE([test for CCDoubles header failed])])

4.1.1 Functions acting on scalar complex values

The C99 language standard library defines common operations on scalar double complex numbers; in addition CCDoubles defines convenience functions to perform the same.

Function: double complex ccdoubles_cplx_mul (double complex O1, double complex O2)

Even though C99 supports the multiplication operation as: 

#include <ccdoubles.h>

double complex  O1 = CCDOUBLES_CPLX(1.2, 3.4);
double complex  O2 = CCDOUBLES_CPLX(5.6, 7.8);
double complex  R  = O1 * O2;

printf("R = %lf%+lfi\n", creal(R), cimag(R));
-| R = -19.800000+28.400000i

#include <ccdoubles.h>

double complex  O1 = CCDOUBLES_CPLX(1.2, 3.4);
double complex  O2 = CCDOUBLES_CPLX(5.6, 7.8);
double complex  R;

R = ccdoubles_cplx_mul(O1, O2);

5.1 Introduction to vector operations

The vectors on which CCDoubles operates are meant to be contiguous arrays of double and double complex values, defined as follows: 

#include <ccdoubles.h>
#undef NSLOTS
#define NSLOTS          2

double          RV[NSLOTS] = { 1.2, 3.4 };
double complex  CV[NSLOTS] = {
  CCDOUBLES_CPLX(1.2, 3.4),
  CCDOUBLES_CPLX(5.6, 7.8)
};

or as follows with dynamic size (which forbids hard–coded initialisation of slots): 

unsigned        nslots = 1;
double          RV[nslots];
double complex  CV[nslots];

Vectors and bidimensional arrays

The memory layout of a vector is equal to the memory layout of the data area of a matrix with a single row defined as follows: 

double                  RM[1][3];
double complex          CM[1][3];

meaning that these definitions are equivalent to: 

double                  RV[3];
double *                RM[1];
RM[0] = &RV[0];

double complex          CV[3];
double complex *        CM[1];
CM[0] = &CV[0];

According to a strict interpretation of the C Language Standard, the definition of a matrix with a single column: 

double                  RM[3][1];
double complex          CM[3][1];

is equivalent to the following: 

double                  RV0[1];
double                  RV1[1];
double                  RV2[1];
double *                RM[3];
RM[0] = &RV0[0];
RM[1] = &RV1[0];
RM[2] = &RV2[0];

double complex          CV0[1];
double complex          CV1[1];
double complex          CV2[1];
double complex *        CM[3];
CM[0] = &CV0[0];
CM[1] = &CV1[0];
CM[2] = &CV2[0];

and there is no guarantee that the memory allocated for the arrays RVx and CVx is contiguous; in pictures, the memory layout is: 

RM
-
| -------> |---| RV0
-
| -------> |---| RV1
-
| -------> |---| RV2
-

and there is no guarantee that it is: 

RM
-          -
| -------> | RV0
-          -
| -------> | RV1
-          -
| -------> | RV2
-          -

Nevertheless, it appears that the GNU C Compiler generates code to allocate a contiguous block of memory when a bidimensional array is defined.

CCDoubles function implementation templates

All the functions have simple implementation involving an iteration over the vector’s slots using restrict pointers; examples for real valued vectors: 

void
ccdoubles_real_vector_add (unsigned nslots,
                           ccdoubles_real_result_t R,
                           ccdoubles_real_operand_t O1,
                           ccdoubles_real_operand_t O2)
{
  for (unsigned i=0; i<nslots; ++i) {
    R[i] = O1[i] + O2[i];
  }
}
void
ccdoubles_real_vector_sin (unsigned nslots,
                           ccdoubles_real_result_t R,
                           ccdoubles_real_operand_t O)
{
  for (unsigned i=0; i<nslots; ++i) {
    R[i] = sin(O[i]);
  }
}

examples for complex valued vectors: 

void
ccdoubles_cplx_vector_add (unsigned nslots,
                           ccdoubles_cplx_result_t R,
                           ccdoubles_cplx_operand_t O1,
                           ccdoubles_cplx_operand_t O2)
{
  for (unsigned i=0; i<nslots; ++i) {
    R[i] = O1[i] + O2[i];
  }
}
void
ccdoubles_cplx_vector_sin (unsigned nslots,
                           ccdoubles_cplx_result_t R,
                           ccdoubles_cplx_operand_t O)
{
  for (unsigned i=0; i<nslots; ++i) {
    R[i] = csin(O[i]);
  }
}

How to call CCDoubles functions

CCDoubles functions can be called as follows, see how the arrays are passed as arguments: 

#include <ccdoubles.h>
#undef NSLOTS
#define NSLOTS         2

double  V[NSLOTS] = { 1.2, 3.4 };
double  R[NSLOTS];

ccdoubles_real_vector_sin(NROWS, NCOLS, R, V);

the result is stored the array referenced by R, overwriting the old contents.

According to the C Language Standard, the arguments R and V in the function call: 

ccdoubles_real_vector_sin(NROWS, NCOLS, R, V);

are implicitly converted to &R[0] and &V[0], pointers to the first element of the arrays.

Function: void ccdoubles_real_vector_linear_combination (unsigned nslots, ccdoubles_real_result_t R, double alpha, ccdoubles_real_operand_t O1, double beta, ccdoubles_real_operand_t O2)

Perform the linear combination of the vector operands and the given scalars. The implementation is: 

for (unsigned i=0; i<nslots; ++i) {
  result[i] = alpha * O1[i] + beta * O2[i];
}
Function: void ccdoubles_real_vector_linspace (unsigned nslots, ccdoubles_real_result_t R, double start, double past)

Fill the result vector with nslots evenly–spaced steps from start to past. It is implemented as follows: 

double        step = (past - start) / ((double)nslots);
for (unsigned i=0; i<nslots; ++i) {
  result[i] = ((double)i) * step + start;
}

As example, the code: 

unsigned    nslots = 10;
double      R[nslots];
double      start = 0.0;
double      past  = 10.0;
ccdoubles_real_vector_linspace (nslots, R, start, past);
ccdoubles_real_vector_print_brackets(stdout, nslots, R);

prints (edited to add newlines): 

[+0.000000 +1.000000 +2.000000 +3.000000
 +4.000000 +5.000000 +6.000000 +7.000000
 +8.000000 +9.000000]

Fill the result vector with nslots logarithmically spaced steps from pow(10.0, start) to pow(10.0, past). It is implemented as follows:

6.1 Introduction to matrix operations

The matrices on which CCDoubles operates are meant to be contiguous arrays of double and double complex values stored in row–major order. Notice that the requirement of contiguous memory is the same enforced by the libraries CBLAS and LAPACKE. For a discussion of row–major versus column–major order allocation see Row–major and column–major order.

We can define matrices as follows: 

#include <ccdoubles.h>
#define NROWS          2
#define NCOLS          2

double          RM[NROWS * NCOLS] = {
  1.2, 3.4,  /* this is 1st row */
  5.6, 7.8   /* this is 2nd row */
};
double complex  CM[NROWS * NCOLS] = {
  CCDOUBLES_CPLX(1.2, 2.3), CCDOUBLES_CPLX(3.4, 4.5),
  CCDOUBLES_CPLX(5.6, 6.7), CCDOUBLES_CPLX(7.8, 8.9)
};

or as follows with variable–length arrays (VLA, which forbid hard–coded initialisation of slots): 

unsigned        nrows = 3;
unsigned        ncols = 4;
double          RM[nrows * ncols];
double complex  CM[nrows * ncols];

If we trust the compiler to allocate contiguous memory for the data area of bidimensional arrays, we can do: 

#include <ccdoubles.h>
#define NROWS          2
#define NCOLS          2

double          RM[NROWS][NCOLS] = {
  { 1.2, 3.4 },  /* this is 1st row */
  { 5.6, 7.8 }   /* this is 2nd row */
};
double complex  CM[NROWS][NCOLS] = {
  { CCDOUBLES_CPLX(1.2, 2.3), CCDOUBLES_CPLX(3.4, 4.5) },
  { CCDOUBLES_CPLX(5.6, 6.7), CCDOUBLES_CPLX(7.8, 8.9) }
};

or as follows with variable–length arrays: 

unsigned        nrows = 3;
unsigned        ncols = 4;
double          RM[nrows][ncols];
double complex  CM[nrows][ncols];

Memory layout of bidimensional arrays

According to a strict interpretation of the C Language Standard, the definition of bidimensional arrays: 

double          RM[3][2];
complex double  CM[3][2];

is equivalent to: 

double          RV0[2];
double          RV1[2];
double          RV3[2];
ccdoubles_real_result_t        RM[3];
RM[0] = &RV0[0];
RM[1] = &RV1[0];
RM[2] = &RV2[0];

double complex   CV0[2];
double complex   CV1[2];
double complex   CV3[2];
double complex * CM[3];
CM[0] = &CV0[0];
CM[1] = &CV1[0];
CM[2] = &CV2[0];

and there is no guarantee that the data area of the arrays RVx and CVx is contiguous; in pictures, the memory layout is: 

RM
-
| -------> |---|---|---| RV0
-
| -------> |---|---|---| RV1
-
| -------> |---|---|---| RV2
-

and there is no guarantee that it is: 

        |    RV0    |    RV1    |    RV2    |
        |---|---|---|---|---|---|---|---|---|
RM        ^           ^           ^
-         |           |           |
| --------            |           |
-                     |           |
| --------------------            |
-                                 |
| --------------------------------
-

Nevertheless, it appears that the GNU C Compiler generates code to allocate a contiguous block of memory when a bidimensional array is defined.

How to call CCDoubles functions

CCDoubles functions can be called as follows, see how the arrays are passed as arguments: 

#include <ccdoubles.h>
#undef NROWS
#undef NCOLS
#define NROWS          2
#define NCOLS          2

double  O[NROWS * NCOLS] = {
  1.2, 3.4,
  5.6, 7.8
};
double  R[NROWS * NCOLS];

ccdoubles_real_matrix_sin(NROWS, NCOLS, R, O);

the result is stored in the array referenced by R, overwriting the old contents. According to the C Language Standard, the arguments R and O in the function call: 

ccdoubles_real_matrix_sin(NROWS, NCOLS, R, O);

are implicitly converted to &R[0] and &O[0], pointers to the first element of the arrays.

If we trust the compiler to allocate contiguous memory for the data area of bidimensional arrays, we can do: 

#include <ccdoubles.h>
#undef NROWS
#undef NCOLS
#define NROWS          2
#define NCOLS          2

double  O[NROWS][NCOLS] = {
  { 1.2, 3.4 },
  { 5.6, 7.8 }
};
double  R[NROWS][NCOLS];

ccdoubles_real_matrix_sin(NROWS, NCOLS, &R[0][0], &M[0][0]);

and we can use the predefined macro MREF() as follows: 

#define CCDOUBLES_ENABLE_SHORT_MACROS   1
#include <ccdoubles.h>

#define NROWS          2
#define NCOLS          2

double  O[NROWS][NCOLS] = {
  { 1.2, 3.4 },
  { 5.6, 7.8 }
};
double  R[NROWS][NCOLS];

ccdoubles_real_matrix_sin(NROWS, NCOLS, MREF(R), MREF(M));

Function: void ccdoubles_real_matrix_linspace (unsigned nrows, unsigned ncols, ccdoubles_real_result_t R, double start, double row_past, double col_past)

Fill the result matrix with nrows times ncols evenly–spaced steps from start to row_past in the horizontal direction and from start to col_past in the vertical direction. It is implemented as follows: 

double        row_step = (row_past - start) / ((double)nrows);
double        col_step = (col_past - start) / ((double)ncols);
for (unsigned i=0; i<nrows; ++i) {
  double      row_first = ((double)i) * row_step + start;
  for (unsigned j=0; j<ncols; ++j) {
    R[i * ncols + j] = ((double)j) * col_step + row_first;
  }
}

7.1 Row–major and column–major order

In this section we will assume the use of a C language compiler that: when allocating bidimensional arrays on the stack, generates code to allocate a contiguous memory region for the data area; this is not guaranteed by the C Language Standard; Introduction to matrix operations.

Row–major and column–major are ways to serialise the contents of a bidimensional array. The mathematical entity matrix: 

 -           -
| a11 a12 a13 |
| a21 a22 a23 |
 -           -

has no ordering by itself; but, by mathematical convention, we indicate its elements as a_{ij}, where i is the row index and j is the column index; the row index first and the column index second.

When we represent such matrix as a bidimensional C language array: the elements are stored in memory in row–major order or in column–major order. As defined by the C Language Standard, the bidimensional array: 

#define M       2
#define N       3
double  A[M][N] = {
  { 1.1, 1.2, 1.3 },
  { 2.1, 2.2, 2.3 }
};

is stored in memory in “first dimension major” order, that is: 

low memory addresses ---> high memory addresses

   1.1     1.2     1.3     2.1     2.2     2.3
|-------|-------|-------|-------|-------|-------|
 A[0][0] A[0][1] A[0][2] A[1][0] A[1][1] A[1][2]

When referencing the elements we write A[i][j], where i is the index of the first dimension and j is the index of the second dimension.

  • If we interpret i as row index and j as column index: the matrix is stored in row–major order.
  • If we interpret i as column index and j as row index: the matrix is stored in column–major order.

In other words, given the mathematical matrix: 

 -           -     -           -
| a11 a12 a13 | = | 1.1 1.2 1.3 |
| a21 a22 a23 |   | 2.1 2.2 2.3 |
 -           -     -           -

we can represent it in row–major order as: 

#define NROWS   2
#define NCOLS   3
double  A[NROWS][NCOLS] = {
  { 1.1, 1.2, 1.3 },
  { 2.1, 2.2, 2.3 }
};

 low memory addresses ---> high memory addresses

   1.1     1.2     1.3     2.1     2.2     2.3
|-------|-------|-------|-------|-------|-------|
 A[0][0] A[0][1] A[0][2] A[1][0] A[1][1] A[1][2]

and in column–major order as: 

#define NROWS   2
#define NCOLS   3
double  A[NCOLS][NROWS] = {
  { 1.1, 2.1 },
  { 1.2, 2.2 },
  { 1.3, 2.3 }
};

 low memory addresses ---> high memory addresses

   1.1     2.1     1.2     2.2     1.3     2.3
|-------|-------|-------|-------|-------|-------|
 A[0][0] A[0][1] A[1][0] A[1][1] A[2][1] A[2][2]

Given a mathematical matrix represented as C language bidimensional array in row–major order: the mathematical operation “matrix transposition” applied to the array transforms the row–major representation into the column–major representation, and vice versa. Beware that, when using complex valued matrices: the usual operation is “matrix transposition and conjugation”, which, obviously, does not have this property.

Ordering under CBLAS and LAPACKE

The libraries BLAS and LAPACK are written in the Fortran language; by language definition: bidimensional arrays in Fortran are represented in column–major order.

The libraries CBLAS and LAPACKE are C language interfaces to BLAS and LAPACK; one of the features of these libraries is to allow the selection of ordering representation by the function parameters:

const enum CBLAS_ORDER Order

For CBLAS. Its values can be CblasRowMajor and CblasColMajor.

int matrix_order

For LAPACKE. Its values can be LAPACK_ROW_MAJOR and LAPACK_COL_MAJOR.

We have seen that, in the C language, we are free to select row–major or column–major representations by simply assigning meaning to the first and second dimensions of a bidimensional array. So we can simply adopt the mathematical convention for the C language representation, define all the matrices as: 

#define NROWS   2
#define NCOLS   3
double  A[NROWS][NCOLS] = {
  { 1.1, 1.2, 1.3 },
  { 2.1, 2.2, 2.3 }
};

and interpret the first dimension as rows and the second dimension as columns. When using CBLAS we will use the argument CblasRowMajor, when using LAPACKE we will use the argument LAPACK_ROW_MAJOR.
