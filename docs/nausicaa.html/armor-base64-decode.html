63.4.3 Decoding byte streams with base64

— Class: <base64-decode-ctx>

Context descriptor for base64 decoding. Nothing in this record depends on the input or output data: records of this type can be reused for different streams, both in sequence and in parallel.

— Immutable Field of <base64-decode-ctx>: encoding

A Scheme symbol among: base64, base64/url. It is the “name” of the table used for decoding.

— Immutable Field of <base64-decode-ctx>: expect-padding?

A Scheme boolean; #t if we expect the encoded input to be padded in length to a multiple of 4.

— Immutable Field of <base64-decode-ctx>: pad-char

An exact integer representing (as defined by char->integer) the expected pad character. It is set to the integer representation of ‘#\=’.

— Immutable Field of <base64-decode-ctx>: table

A Scheme vector of exact integers representing the selected decoding's table.

— Function: make-<base64-decode-ctx> encoding expect-padding?

Build and return a new record of type <base64-decode-ctx>.

encoding must be a Scheme symbol selecting the decoding table: base64, base64/url.

If expect-padding? is true: the input is expected to be padded in length to an exact multiple of 4. The argument's value is normalised to #t or #f.

— Function: <base64-decode-ctx>? obj

Return true if obj is a record of type <base64-decode-ctx>.

— Function: <base64-decode-ctx>-encoding ctx
— Function: <base64-decode-ctx>-expect-padding? ctx
— Function: <base64-decode-ctx>-pad-char ctx
— Function: <base64-decode-ctx>-table ctx

Field accessors for records of type <base64-decode-ctx>.

— Function: base64-decode-update-length number-of-chars

Return the minimum number of bytes required in the output bytevector when base64-decode-update! is applied to number-of-chars ASCII characters of input. Notice that the function will consume input in 4-bytes blocks only and produce output in 3-bytes blocks only, so the return value is always zero or an exact multiple of 3.

— Function: base64-decode-final-length number-of-chars
— Function: base64-decode-final-length number-of-chars padding?

Return the minimum number of bytes required in the output bytevector when base64-decode-final! is applied to number-of-chars < 4 ASCII characters of input. Not all the arguments between 0 and 3 are valid; if number-of-chars is not valid: the return value is #f.

If padding? is true: the input is expected to be padded; in this case the only valid value for number-of-chars is zero and the return value is zero. If padding? is true and number-of-chars is non–zero: the return value is #f. When not given, padding? defaults to #f.

— Function: base64-decode-length number-of-chars
— Function: base64-decode-length number-of-chars padding?

Return the minimum number of bytes required in the output bytevector when base64-decode-final! is applied to number-of-chars ASCII characters of input. This is also the number required when both base64-decode-update! and base64-decode-final! are used.

If padding? is true, the input is expected to be padded; in this case the only valid value for number-of-chars is zero and the return value is zero. If padding? is true and number-of-chars is non–zero, the return value is #f. When not given, padding? defaults to #f.

— Constant: base64-decode-block-length

The length of the input block: 4.

— Function: base64-decode-update! ctx dst-bv dst-start src-bv src-start src-past

Decode ASCII characters from the bytevector src-bv starting at index src-start (included) up to index src-past (excluded); store binary data bytes in the bytevector dst-bv starting at dst-start (included); decoding is performed according to the specification in the context ctx, which must be a record of type <base64-decode-ctx>.

Return three values:

  1. A boolean: true if padding is turned on and a full padded block was successfully decoded; false otherwise. If this is true: the end of input has been reached and fully processed.
  2. The index of the next non–written byte in dst-bv; if dst-bv is filled to the end, this value is the length of dst-bv.
  3. The index of the next non–read byte in src-bv; if all the characters from src-bv are consumed, this value is src-past.

ASCII characters are read from src-bv in chunks of 4 bytes; binary data bytes are written to dst-bv in chunks of 3 bytes. If not enough room exists in dst-bv to store a block, available input data is not consumed.

— Function: base64-decode-final! ctx dst-bv dst-start src-bv src-start src-past

Decode ASCII characters from the bytevector src-bv starting at index src-start (included) up to index src-past (excluded); store binary data bytes in the bytevector dst-bv starting at dst-start (included); decoding is performed according to the specification in the context ctx, which must be a record of type <base64-decode-ctx>.

Return three values:

  1. A boolean: true if success, false if the output bytevector was filled before flushing all the bytes.
  2. The index of the next non–written byte in dst-bv; if dst-bv is filled to the end, this value is the length of dst-bv.
  3. The index of the next non–read byte in src-bv; if all the bytes from src-bv are consumed, this value is src-past.

ASCII characters are read from src-bv in chunks of 4 bytes; binary data bytes are written to dst-bv in chunks of 3 bytes. If the last characters are in a chunk of less than 4, they are interpreted as unpadded characters as specified in ctx. If not enough room exists in dst-bv to store a block, available input data is not consumed.
