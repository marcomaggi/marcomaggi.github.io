11.3.1.6 Error recovering

Until now we have seen only examples for which, given a sequence of symbols ‘S A B C’ and a lookahead ‘D’, at least one of the sequences: 

     S A B C D
     A B C D
     B C D
     C D

has an action associated to it in the grammar; in other words, using the graph model:

  • The state associated to the sequence ‘S A B C’ has an outgoing link associated to the lookahead ‘D’, or
  • The state associated to the sequence ‘A B C’ has an outgoing link associated to the lookahead ‘D’, or
  • The state associated to the sequence ‘B C’ has an outgoing link associated to the lookahead ‘D’, or
  • The state associated to the sequence ‘C’ has an outgoing link associated to the lookahead ‘D’.

What happens if the lookahead is not associated to an outgoing link? It means that the sequence of tokens does not comply with the defined grammar. There are two possible reactions to this: stop parsing and signal an error, or try to recover.

Recovering makes sense only when the grammar is such that: it is possible to discard a subsequence of symbols as invalid, and go on parsing the following symbols.

In the example of the single expressions parser: if the lookahead returns an invalid category symbol, the only possible reaction is to stop parsing and signal an error. The whole expression is wrong.

In the example of the script expression parser: we can discard an invalid line and try to parse the following ones. Let's examine how this is done by the LR driver.

The valid terminal categories are: 

     N A M O C T *eoi*

where ‘T’ is the line terminator; examples of valid lines are: 

     N T
     A N T
     N A N T
     N M N T
     N A N M N T

If we use a grammar describing a sequence of lines, the grammar “knows” that after each line it has to reduce the stacks to the starting symbol; it knows that ‘S’ is the first symbol of each line. For example: 

     #(#f  1  +  2  #\newline  3  *  4  #\newline  #<eof>)
     #(S   N  A  N  T          N  M  N  T          *eoi*)

after parsing the first line is reduced to: 

     #(#f  3  *  4  #\newline  #<eof>  #f #f #f)
     #(S   N  M  N  T          *eoi*   #f #f #f)

When writing the grammar specification: we associate an error action to the beginning of a line, and it ends up being associated to ‘S’.

Let's consider the following couple of lines, with the first being a wrong expression: 

     #(#f  1  +  #\newline  3  *  4  #\newline  #<eof>)
     #(S   N  A  T          N  M  N  T          *eoi*)

when the stack pointer references the symbol ‘A’, the lookahead returns ‘T’ and the grammar signals an error. The parser decrements the pointer, “rewinding” the stack, in search of a symbol which has an error action associated to it. It finds that ‘S’ has such an action, and it is associated to the symbol ‘T’; so it removes all the symbols between ‘S’, excluded, and the first ‘T’, excluded: 

     #(#f  #\newline  3  *  4  #\newline  #<eof>  #f #f)
     #(S   T          N  M  N  T           *eoi*  #f #f)
       |
       p

and it restarts parsing doing a lookahead that returns ‘T’; the action is a shift: 

     #(#f  #\newline  3  *  4  #\newline  #<eof>  #f #f)
     #(S   T          N  M  N  T           *eoi*  #f #f)
           |
           p

the lookahead returns ‘N’ and the action is a reduce: 

     #(#f  3  *  4  #\newline  #<eof>  #f #f #f)
     #(S   N  M  N  T          *eoi*   #f #f #f)
       |
       p

we have recovered from the error.
