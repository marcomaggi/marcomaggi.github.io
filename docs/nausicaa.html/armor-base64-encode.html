66.4.2 Encoding byte streams with base64

— Class: <base64-encode-ctx>

Context descriptor for base64 encoding. Nothing in this record depends on the input or output data: records of this type can be reused for different streams, both in sequence and in parallel.

— Immutable Field of <base64-encode-ctx>: encoding

A Scheme symbol among: base64, base64/url. It is the “name” of the table used for encoding.

— Immutable Field of <base64-encode-ctx>: generate-padding?

A Scheme boolean value; #t if the output data is padded so that its total length in bytes is a multiple of 4.

— Immutable Field of <base64-encode-ctx>: pad-char

An exact integer representing (as defined by char->integer) the pad character. It is set to the integer representation of ‘#\=’.

— Immutable Field of <base64-encode-ctx>: table

A Scheme vector of exact integers representing the selected encoding's alphabet.

— Function: make-<base64-encode-ctx> encoding generate-padding?

Build and return a new record of type <base64-encode-ctx>.

encoding must be a Scheme symbol selecting the encoding type: base64, base64/url.

If generate-padding? is true: the output is padded so that its length is an exact multiple of 4. The argument's value is normalised to #t or #f.

— Function: <base64-encode-ctx>? obj

Return true if obj is a record of type <base64-encode-ctx>.

— Function: <base64-encode-ctx>-encoding ctx
— Function: <base64-encode-ctx>-generate-padding? ctx
— Function: <base64-encode-ctx>-pad-char ctx
— Function: <base64-encode-ctx>-table ctx

Field accessors for records of type <base64-encode-ctx>.

— Function: base64-encode-update-length number-of-bytes

Return the minimum number of bytes required in the output bytevector when base64-encode-update! is applied to number-of-bytes of binary input data. Notice that that function will consume input in 3-bytes blocks only and produce output in 4-bytes blocks only, so the return value is always zero or an exact multiple of 4.

— Function: base64-encode-final-length number-of-bytes
— Function: base64-encode-final-length number-of-bytes padding?

Return the minimum number of bytes required in the output bytevector when base64-encode-final! is applied to number-of-bytes < 3 bytes of binary input data.

If padding? is true: the returned count includes padding bytes, which means that the returned value is always 4; when not used, padding? defaults to #f.

If number-of-bytes is invalid (out of range): the return value is #f.

— Function: base64-encode-length number-of-bytes
— Function: base64-encode-length number-of-bytes padding?

Return the minimum number of bytes required in the output bytevector when base64-encode-final! is applied to number-of-bytes of binary input data. This is also the number required when both base64-encode-update! and base64-encode-final! are used.

If padding? is true: the returned count includes padding bytes, which means that the returned value is always zero or a multiple of 4; when not used, padding? defaults to #f.

— Constant: base64-encode-block-length

The length of the input block: 3.

— Function: base64-encode-update! ctx dst-bv dst-start src-bv src-start src-past

Encode binary data from the bytevector src-bv starting at index src-start (included) up to index src-past (excluded); store ASCII characters in the bytevector dst-bv starting at dst-start (included); encoding is performed according to the specification in the context ctx, which must be a record of type <base64-encode-ctx>.

This function is meant to encode bytes from the input stream when we know that more input bytes will be available in the future.

Return two values:

  1. The index of the next non–written byte in dst-bv; if dst-bv is left full, this value is the length of dst-bv.
  2. The index of the next non–read byte in src-bv; if all the bytes from src-bv are consumed, this value is src-past.

Binary data bytes are read from src-bv in chunks of 3 bytes; ASCII characters are written to dst-bv in chunks of 4 bytes. If not enough room exists in dst-bv to store a block, available input data is not consumed.

— Function: base64-encode-final! ctx dst-bv dst-start src-bv src-start src-past

Encode binary data from the bytevector src-bv starting at index src-start (included) up to index src-past (excluded); store ASCII characters in the bytevector dst-bv starting at dst-start (included); encoding is performed according to the specification in the context ctx, which must be a record of type <base64-encode-ctx>.

This function is meant to encode the last bytes from the input stream and produce the last bytes to the output stream.

Return three values:

  1. A boolean: true if success, false if the output bytevector was filled before flushing all the bytes.
  2. The index of the next non–written byte in dst-bv; if dst-bv is left full, this value is the length of dst-bv.
  3. The index of the next non–read byte in src-bv; if all the bytes from src-bv are consumed, this value is src-past.

Binary data bytes are read from src-bv in chunks of 3 bytes; ASCII characters are written to dst-bv in chunks of 4 bytes. After that, trailing bytes from src-bv are encoded and optionally padded as specified in ctx. If not enough room exists in dst-bv to store a block, available input data is not consumed.
