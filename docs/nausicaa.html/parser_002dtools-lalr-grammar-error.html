12.3.5.5 Error recovery
Introduction to error recovery

(nausicaa parser-tools lalr) implements a very simple error recovery strategy. A production can be of the form: 

     (NON-TERMINAL
        ---                  ;right-hand side rules
        (error TERMINAL) : semantic-clause)

If, while parsing the right–hand side rules for ‘NON-TERMINAL’, the lookahead returns a token of invalid category: the parser resets the state associated with the parsing of the rule and skips all the tokens returned by the lexer until one with category ‘TERMINAL’ is found. The ‘TERMINAL’ token is also discarded and parsing continues with a new lookahead.

The symbol ‘error’ is reserved as error recovery directive, it cannot be used as terminal or non–terminal.

There can be several such right–hand side rules for a single non–terminal: for a C–like language, we could “synchronise” on semicolons and closing curly brackets by writing error rules like these: 

     (statement
        (expression SEMICOLON)        : ---
        (LBRACKET statement RBRACKET) : ---
        (error SEMICOLON)    ;empty semantic clause
        (error RBRACKET))    ;empty semantic clause

if an error occurs, the parser will discard its internal state up to, and including, the first token that started the ‘statement’ non–terminal (‘LBRACKET’ or the first of ‘expression’).

Dummy example: Single token parser

Let's consider the following parser which accepts as input only the sequence: ‘A’, ‘*eoi*’; it is a corner case which is useful to understand. The correct way to handle this case is to have an error handler that does not return; in the following the error handler returns, leaving the parser to handle everything. 

     (define make-parser
       (lalr.lalr-parser
         (lalr.output-value: #t)
         (lalr.terminals:    '(A))
         (lalr.rules:
          '((e (A) : $1)))))

We have already examined the correct usage of this grammar (Dummy examples of grammar definitions); we can now understand what happens in the following test: 

     (example (make-token 'A 1)
              (make-token 'A 2)
              (make-token 'A 3)))
     ⇒ ("unexpected end of input" . (eof-object))

following the reasoning exposed in the introduction (see Introduction to the LR parser), we can imagine the input tokens on the following stacks: 

     #(#f  1  2  3  #<eof>)
     #(S   A  A  A   *eoi*)
       |
       p
  1. The first lookahead returns ‘A’ and the action is a shift: 
              #(#f  1  2  3  #<eof>)
          #(S   A  A  A   *eoi*)
                |
                p
  2. The second lookahead returns ‘A’ and the action is a reduce, which changes nothing because there is only one token and the semantic action is ‘$1’: 
              #(#f  1  2  3  #<eof>)
          #(S   A  A  A   *eoi*)
                |
                p
  3. The third lookahead returns ‘A’ and the action is error because only ‘*eoi*’ is acceptable now. The error handler is invoked and its return value discarded.

    The parser attempts to recover from the error; we already know that the recovery will fail, because we have not included any ‘error’ right–hand side rule in the grammar.

  4. The first operation of recovery is to reset the current parser state removing everything related to this right–hand side rule: 
              #(#f  2  3  #<eof>  #f)
          #(S   A  A   *eoi*  #f)
            |
            p
  5. The parser detects the fact that the stack has been fully rewind without finding a non–terminal with an ‘error’ directive, so it just stops reading tokens from the lexer and imposes a ‘*eoi*’ token as lookahead.
  6. The end–of–input token is invalid as lookahead from the initial state, so an “unexpected end–of–input error” is generated; the error handler is invoked and its return value becomes the return value of the parser closure.
