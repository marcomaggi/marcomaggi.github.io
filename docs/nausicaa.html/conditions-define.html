8.1 Defining new condition types

The following bindings are exported by the library (nausicaa language conditions).

— Syntax: define-condition-type <condition-type> <supertype> <constructor> <predicate> (<field> <accessor>) ...

A transparent replacement for define-condition-type from (rnrs conditions (6)) which defines a new condition object type with additional features to better merge it in to the Nausicaa language, see define-condition-type.

It can be used exactly like the original, but we cannot mix the hierarchy of <condition-type> identifiers created by the original with the hierarchy of identifiers created by this replacement.

This syntax takes advantage of the fact that, in normal code using (rnrs conditions (6)), the identifier of a condition type is used only in define-condition-type forms; so this syntax defines a condition type using define-condition-type from (rnrs conditions (6)), assigning to it an automatically generated name ‘the-type’: 

          (rnrs.define-condition-type the-type
              <supertype>
            <constructor>
            <predicate>
            (<field> <accessor>)
            ...)

then it defines a label using the type name as label name and shadowing ‘the-type’: 

          (define-label <condition-type>
            (shadows the-type)
            (custom-maker <constructor>)
            (predicate <predicate>)
            (virtual-fields (immutable <field> <accessor>)
                            ...))

the result is a condition type definition which can be used as usual inside a hierarchy of types created by this syntax; in addition we have the features of a label, that is <condition-type> can be used by is-a?, with-class and derived syntaxes.

Condition objects of types created by this syntax cannot be used to dispatch generic functions as defined by (nausicaa language generics).

— Syntax: define-condition <condition-type> <clause> ...
— Auxiliary Syntax: parent <supertype>
— Auxiliary Syntax: fields <field-spec> ...

Define a new condition object type <condition-type>; the name of the identifier <condition-type> must have ‘&’ as first character; this syntax has the same features of define-condition-type described above, but in addition it allows fields tagging. As example, the definition: 

          (define-condition &wrong-num-args
            (parent &assertion)
            (fields procname expected given))

is equivalent to: 

          (define-condition-type &wrong-num-args
              &assertion
            make-wrong-num-args-condition
            wrong-num-args-condition?
            (procname     condition-wrong-num-args/procname)
            (expected     condition-wrong-num-args/expected)
            (given        condition-wrong-num-args/given))

in which we see several automatically generated identifiers.

— Auxiliary Syntax: parent <parent-type>

Select the parent condition type for the enclosing definition. This clause can be omitted and can be used at most once. When not used, the parent defaults to &condition.

<parent-type> must be an identifier defined by define-condition-type described above.

The parent identifier is the one exported by (rnrs syntactic (6)) or (nausicaa language classes auxiliary-syntaxes) for the implementations that do not define it.

— Auxiliary Syntax: fields <field-spec> ...

Specify the fields of the enclosing condition type definition. All the <field-spec> arguments must have one of the forms: 

          <field>
          (<field> <accessor>)
          ((<field> <tag> ...))
          ((<field> <tag> ...) <accessor>)

in which: the identifier <field> is the name of the field; the identifier <accessor> is the name of the accessor of the field; the identifier <tag> is the name of a label or class which tags the field (classes define tagged). This clause can be omitted and can be used at most once.

When the first form of <field-spec> is used: for each field an accessor name is built as follows: the ‘&’ character is stripped from <condition-type>, a slash is appended to the result, the field name is appended to the result, the string ‘condition-’ is prepended to the result.

The fields identifier is the one exported by (rnrs syntactic (6)) or (nausicaa language classes auxiliary-syntaxes) for the implementations that do not define it.
