31.1.8 Accessing data structures

We examine a couple of real–world cases of C language data structure access, with the strategy proposed by the Nausicaa distribution. for each field we want to access in a foreign data structure, we need to know:

  • The offset of the first byte of the field in the structure's memory block.
  • The C language data type of the field. Knowing the data type is equivalent to knowing the required peeker and poker. Direct access to memory blocks

Using the Nausicaa foreign library inspection infrastructure, we can put the following in GNU Autoconf's configure.ac template of a Zlib binding: 

     m4_define([NAUSICAA_CUSTOM_INCLUDES],[
     #include <zlib.h>
     ])
     
     NAUSICAA_INSPECT_STRUCT_TYPE([Z_STREAM],[z_stream],[#f])
     NAUSICAA_INSPECT_FIELD_TYPE([Z_STREAM_NEXT_IN],
       [z_stream],[next_in],[pointer])
     NAUSICAA_INSPECT_FIELD_TYPE([Z_STREAM_AVAIL_IN],
       [z_stream],[avail_in],[unsigned-int])
     NAUSICAA_INSPECT_FIELD_TYPE([Z_STREAM_TOTAL_IN],
       [z_stream],[total_in],[unsigned-int])

and the following in a library to be processed by the configure script: 

     (define typeof-z_stream.next_in   '@TYPEOF_Z_STREAM_NEXT_IN@)
     (define offsetof-z_stream.next_in  @OFFSETOF_Z_STREAM_NEXT_IN@)
     
     (define typeof-z_stream.avail_in   '@TYPEOF_Z_STREAM_AVAIL_IN@)
     (define offsetof-z_stream.avail_in @OFFSETOF_Z_STREAM_AVAIL_IN@)
     
     (define typeof-z_stream.total_in   '@TYPEOF_Z_STREAM_TOTAL_IN@)
     (define offsetof-z_stream.total_in @OFFSETOF_Z_STREAM_TOTAL_IN@)

which, after processing, becomes: 

     (define typeof-z_stream.next_in    'pointer)
     (define offsetof-z_stream.next_in  0)
     
     (define typeof-z_stream.avail_in   'unsigned-int)
     (define offsetof-z_stream.avail_in 4)
     
     (define typeof-z_stream.total_in   'unsigned-int)
     (define offsetof-z_stream.total_in 8)

now it is easy to define field accessors and mutators as functions: 

     (define (struct-z_stream.next_in-set! pointer value)
       (pointer-c-set! typeof-z_stream.next_in pointer
                       offsetof-z_stream.next_in value))
     
     (define (struct-z_stream.next_in-ref pointer)
       (pointer-c-ref  typeof-z_stream.next_in pointer
                       offsetof-z_stream.next_in))
     
     ...

or as macros: 

     (define-syntax struct-z_stream.next_in-set!
       (syntax-rules ()
         ((_ ?pointer ?value)
          (pointer-c-set! typeof-z_stream.next_in ?pointer
                          offsetof-z_stream.next_in ?value))))
     
     (define-syntax struct-z_stream.next_in-ref
       (syntax-rules ()
         ((_ ?pointer)
          (pointer-c-ref  typeof-z_stream.next_in ?pointer
                          offsetof-z_stream.next_in))))
     
     ...

a user–friendly way to use these accessors and mutators is to define a label with the features of the (nausicaa language classes) library: 

     (define-label <struct-z_stream>
       (virtual-fields
         (mutable next_in
                  struct-z_stream.next_in-set!
                  struct-z_stream.next_in-ref)
         (mutable avail_in
                  struct-z_stream.avail_in-set!
                  struct-z_stream.avail_in-ref)
         (mutable total_in
                  struct-z_stream.total_in-set!
                  struct-z_stream.total_in-ref)))

now we can do: 

     (let/with-class
         (((zstream <struct-z_stream>) (malloc (c-sizeof z_stream))))
     
       (set! zstream.next_in  ---)
       (set! zstream.avail_in ---)
     
       ---)

if we use the macro versions of the structure accessors and mutators: the set! forms will be expanded to the direct use of the low level accessor and mutator peeker and poker functions.

Occasionally, a C language data structure embeds an array of values; this is the case of ‘struct dirent’ defined by POSIX, which may be declared like this (from the GNU C library): 

     struct dirent
     {
       __ino_t               d_ino;
       __off_t               d_off;
       unsigned short int    d_reclen;
       unsigned char         d_type;
       char                  d_name[256];
     };

the field ‘d_name’ cannot be accessed like the other fields; what we need to do is to take the offset of the first byte of ‘d_name’ and add it to the structure's pointer: the result is a pointer to the first byte in the embedded array.

Using the Nausicaa foreign library inspection infrastructure, we can put the following in GNU Autoconf's configure.ac template of a POSIX binding: 

     m4_define([NAUSICAA_CUSTOM_INCLUDES],[
     #include <dirent.h>
     ])
     
     NAUSICAA_INSPECT_STRUCT_TYPE([DIRENT],[struct dirent],[#f])
     NAUSICAA_INSPECT_FIELD_TYPE_POINTER([DIRENT_D_NAME],
       [struct dirent],[d_name])

and the following in a library to be processed by the configure script: 

     (define typeof-dirent.d_name   '@TYPEOF_DIRENT_D_NAME@)
     (define offsetof-dirent.d_name  @OFFSETOF_DIRENT_D_NAME@)

which, after processing, becomes: 

     (define typeof-dirent.d_name   'pointer)
     (define offsetof-dirent.d_name 11)

now it is easy to define the field accessor returning a pointer to the first byte of the field, as function: 

     (define (struct-dirent.d_name-ref pointer)
       (pointer-add pointer offsetof-dirent.d_name))

or as macro: 

     (define-syntax struct-dirent.d_name-ref
       (syntax-rules ()
         ((_ ?pointer)
          (pointer-add ?pointer offsetof-dirent.d_name))))

a user–friendly way to use this accessor is to define a label with the features of the (nausicaa language classes) library: 

     (define-label <struct-dirent>
       (virtual-fields
         (immutable d_name struct-dirent.d_name-ref)))

to be used with a POSIX binding library as: 

     (let*/with-class
         ((dir (opendir "/home/marco"))
          ((entry <struct-dirent>) (readdir dir)))
     
       entry.d_name  ;pointer to a C string
     
       ---)
