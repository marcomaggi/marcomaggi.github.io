11.4.1.1 Tokenising an IPv4 address string

The following bindings are exported by the library (nausicaa parser-tools ipv4-addresses).

— Constant: ipv4-address-lexer-table

A lexer table to be used by the library (vicare parser-tools silex lexer) to tokenise IPv4 address or address prefix textual representations; the table is prepared to make use of all the counters. The token categories are:

DOT
Hold as semantic value the character #\..
NUMBER
Hold as semantic value a number in the input IPv4 address.
PREFIX-LENGTH
Hold as semantic value the number of bits in the prefix as exact integer.

The longest IPv4 address string has: 4 numbers as address components, 3 separators between address components, a final slash and number to specify the address prefix length; the total is 4 + 3 + 1 = 8 tokens. We may want to use make-max-count-lexer from (vicare parser-tools silex utilities) to limit the number of returned tokens.

To tokenise an IPv4 address string we can do: 

          (import (nausicaa)
            (prefix (vicare parser-tools silex lexer)        lex.)
            (prefix (nausicaa parser-tools ipv4-addresses)   ip.)
            (prefix (nausicaa parser-tools lexical-tokens)   lt.)
            (prefix (nausicaa parser-tools source-locations) sl.))
          
          (define (make-address-lexer address-string)
            (let ((IS (lex.make-IS (lex.string: address-string))))
              (lex.make-lexer ip.ipv4-address-lexer-table IS)))

every time the lexer thunk is evaluated, it returns an object of type <lexical-token> as defined by the library (nausicaa parser-tools lexical-tokens), Describing lexical tokens. Here are some examples of tokenisation: 

          "1.2.3.4"
          ⇒ [NUMBER 1] [DOT .] [NUMBER 2] [DOT .]
             [NUMBER 3] [DOT .] [NUMBER 4] [*eoi* #<eof>]
          
          "1.Zciao"
          ⇒ [NUMBER 1] [DOT .] [*lexer-error* "Zciao"]

notice that the lexer will match numbers in the range [0, 255] as a single NUMBER token, but positive numbers outside this range are split into multiple tokens: 

          "256" ⇒ [NUMBER 25] [NUMBER 6] [*eoi* #<eof>]
          
          "500" ⇒ [NUMBER 50] [NUMBER 0] [*eoi* #<eof>]

this is weird, but this is just a lexer: it is responsibility of the parser to recognise invalid sequences of tokens.

— Syntax: make-ipv4-address-lexer ?clause ...
— Auxiliary Syntax: string: ?string
— Auxiliary Syntax: port: ?port
— Auxiliary Syntax: procedure: ?proc

Build and return a SILex lexer function making use of the table ipv4-address-lexer-table; the input system is created as described by the given clauses; the table and the input system are prepared to make use of all the counters.

The auxiliary syntaxes are the ones exported by (vicare parser-tools silex lexer); their arguments are the same as the ones described for make-IS, Input systems.
