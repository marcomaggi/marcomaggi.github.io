58.4.7 The grammar combinator

— Syntax: make-grammar-combinator <start> <nonterminal> ...

Provide syntactic sugar for building complex parser combinators from simpler combinators. Each <nonterminal> must be like this: 

          <nonterminal> -> (<identifier> <rule>*)
          
          <rule> -> (<sequence> <semantic-clause>)
          
          <sequence> -> (<part>*)
                     |  (<part>* :error <message>)
                     |  (<part>* :exception)
          
          <part> -> (:not <part>*)
                 |  <var> <- (quote <terminal>)
                 |  <var> <- <identifier>
                 |  <var> <- :source-location
                 |  (quote <terminal>)
                 |  <identifier>

and it expands into a combinator: 

          (make-memoize-combinator (lambda (state) ---))

each non–terminal combinator is bound by a letrec* form to the respective <identifier>: 

          (letrec* ((<identifier> (make-memoize-combinator ---))
                    ...)
            <start>)

the last expression in the body of the generated letrec* is <start>, which can be any expression but must evaluate to a combinator function. The simplest <start> is an <identifier>, which causes the macro to return the combinator bound to it.

Since begin forms within letrec* forms are flattened (spliced into the <body>, Bodies): By making <start> a begin form we can introduce bindings to hand–written combinators which can call (and can be called by) the non–terminal combinators.

Each <sequence> expands into a combinator; the <sequence> combinators of a non–terminal combinator, are all alternatives for its matching.

When a non–terminal combinator is applied to a <state> record, it attempts to match the input tokens against the <sequence> combinators in its <rule> elements. The first <rule> is tried first, then the second and so on, stopping at the first match. If matching is successful, the <semantic-clause> is evaluated to produce a value returned as field of a <success> record; else an <error> record is returned.

Each <part> expands to a combinator which, if successful, invokes the combinator generated by its subsequent <part>; the first <part> combinator of a <sequence> is invoked by the <sequence> combinator; the last <part> combinator of a <sequence> must generate a <result> record by itself.

If a <part> combinator, which is not the last in a <sequence>, matches one or more tokens: It must call next-state to generate a new <state> record and hand it to the next <part> combinator.

If all the <part> combinators in a <sequence> match the tokens, a <success> record is generated by the last one and returned. The <success> record is registered in the memoized results of the <state> record that started a non–terminal match, using <identifier> as keyword.

If a <sequence> combinator matches, but its last <part> is ‘:error’, an <error> record is returned and registered in the memoized results of the <state> record that started the non–terminal match.

If a <sequence> matches, but its last <part> is ‘:exception’, a continuable exception is raised.

If a <part> fails to match: An <error> record is generated and returned. The <error> record is registered in the memoized results of the <state> record that started the non–terminal mismatch.

The <part> elements can be of the following forms.

‘(quote <terminal>)’
Matches if the lookahead token has category eq? to <terminal>; the semantic value of the token is discarded. Consume one token. This <part> makes use of a combinator generated by make-terminal-combinator.
‘<var> <- (quote <terminal>)’
Matches if the lookahead token has category eq? to <terminal>; bind the token's semantic value to <var> and continue parsing; the subsequent combinators are evaluated in the augmented environment. Consume one token. This <part> makes use of a combinator generated by make-terminal-combinator.
‘<identifier>’
Apply the combinator bound to <identifier> to the current state. The applied combinator may consume any number of tokens.
‘<var> <- <identifier>’
Apply the combinator bound to <identifier> to the current state. If successful, bind the resulting semantic value to <var>; the subsequent combinators are evaluated in the augmented environment. The applied combinator may consume any number of tokens.
‘(:not <fail-part> ...)’
Matches if the lookahead tokens do not match the <fail-part> elements. This <part> makes use of a combinator generated by make-unless-combinator.
‘<var> <- :source-location’
Bind the <source-location> record in the lookahead token to <var>; the subsequent combinators are evaluated in the augmented environement. Consumes no tokens. This can be used to annotate abstract syntax trees with source location informations.
‘:error <message>’
Matches only as last <part> of a <sequence> and consumes no tokens. Return an <error> record using <message> as value for the ‘message’ field.
‘:exception’
Matches only as last <part> of a <sequence> and consumes no tokens. The <semantic-clause> of the <rule> is evaluated and it must return a condition object, which is handed to raise-continuable. If the exception handler returns, it must return a <result> record. Conditions
