67.2.2 Lexing JSON text

The following bindings are exported by the (nausicaa json) library.

— Function: make-json-rfc-lexer input-system

Build and return a (nausicaa silex) lexer for JSON text using input-system; the lexer will parse JSON text conforming to RFC 4627.

Each time the lexer function is invoked, it returns a <lexical-token> record with category in the following set of symbols: 

          BEGIN_ARRAY             END_ARRAY
          BEGIN_OBJECT            END_OBJECT
          NAME_SEPARATOR          VALUE_SEPARATOR
          
          FALSE TRUE NULL NUMBER STRING

with the addition of *eoi* for end–of–input and *lexer-error* for invalid input. The semantic values in the records are the JSON values converted to Scheme values: strings, numbers, true, false, null, and characters for the JSON structural characters.

When a token with category *lexer-error* is returned: its semantic value is set to the offending fragment from the input; at most 10 characters from the input are included, followed by an ellipsis if the fragment is not at the end of input. For example the following JSON object: 

          { \"Count\" . 12, \"ciao\": false }

has the dot as first offending character, the *lexer-error* token will have the string " . 12, \"ci..." as semantic value.

— Function: make-json-extended-lexer input-system

Like make-json-rfc-lexer but supports the following non–RFC extensions:

  • The literal ‘Infinity’ is decoded as the positive infinity, ‘+inf.0’.
  • The literal ‘-Infinity’ is decoded as the negative infinity, ‘-inf.0’.
  • The literal ‘NaN’ is decoded as not–a–number, ‘+nan.0’.
  • Numbers are accepted in binary, octal and hexadecimal format when they are prefixed, respectively, by: ‘#b’ or ‘#B’, ‘#o’ or ‘#O’, ‘#x’ or ‘#X’. Decoding is performed by number->string.

— Function: json->tokens input-system

Read JSON text from input-system and return a list of <lexical-token> records. Processing of input stops at the first token with category *eoi* or *lexer-error*.
