8.2 Handling exceptions

The following bindings are exported by the library (nausicaa language conditions).

— Syntax: guard/with-class (?variable ?cond-clause1 ?cond clause ...) ?body
— Auxiliary Syntax: =>
— Auxiliary Syntax: else
— Auxiliary Syntax: is-a?
— Auxiliary Syntax: with-class

A replacement for guard from (rnrs exceptions (6)) which can be used in the same way, see guard.

The auxiliary syntaxes => and else are the identifiers exported by (rnrs base (6)) and they are reexported by (nausicaa language conditions); the auxiliary syntaxes is-a? and with-class are the identifiers exported by (nausicaa language classes) and they are reexported by (nausicaa language conditions).

The ?cond-clause components must have the form: 

          (?test ?expression ...)
          (?test => ?expression ...)
          (else ?expression1 ?expression ...)

This syntax recognises as special some ?test expressions appearing in the first form of ?cond-clause:

((is-a? ?condition-type ...) ?expression ...)
The ?condition-type components must be condition type names as defined by define-condition-type from (nausicaa language conditions).

The ?test evaluates to true if ?variable is bound to a condition object having the listed ?condition-type as components; the test is equivalent to: 

               (and (is-a? ?variable ?condition-type)
                    ...)

the ?expression components are wrapped in a with-class allowing access to condition object's fields: 

               (with-class ((?variable ?condition-type ...))
                 ?expression
                 ...)

Notice that a ?test with an empty list of ?condition-type components is always false, so such a clause is recognised at expand time and removed: this means that side effects of ?expression components' expansion are not produced.

((is-a? ?condition-type ... (with-class ?tag ...)) ?expression ...)
The ?condition-type components must be condition type names as defined by define-condition-type from (nausicaa language conditions), the ?tag components can be condition type names or class names or label names as defined by (nausicaa language classes).

The ?test evaluates to true if ?variable is bound to a condition object having the listed ?condition-type as components; the test is equivalent to: 

               (and (is-a? ?variable ?condition-type)
                    ...)

the ?expression components are wrapped in a with-class allowing access to condition object's fields and ?tag fields: 

               (with-class ((?variable ?tag ...)
                            (?variable ?condition-type ...))
                 ?expression
                 ...)

Notice that a ?test with an empty list of ?condition-type is always false, so such a clause is recognised at expand time and removed: this means that side effects of ?expression components' expansion are not produced.

This syntax is useful when we know that the presence of the ?condition-type components, in a compound condition object, implies the presence of the ?tag components.
