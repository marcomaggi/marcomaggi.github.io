23.1.7.1 Dynamic loading of libraries

A foreign shared library loadable with the dlopen() POSIX interface is called shared object. Under Nausicaa/Scheme, a shared object can be opened, but not closed. The API makes use of a Scheme object to represent a shared object; it allows access to all the functions and variables exported by the library. The following bindings are exported by the (ffi) library.

— Function: open-shared-object library-name

Open a foreign shared library on the host system and return a shared object; shared objects are disjoint from the other Scheme object types.

If opening the shared library fails: throw a continuable exception with raise-shared-object-opening-error. Shared objects opening error.

library-name must be a Scheme string or symbol representing the file name or pathname to be opened; examples: ‘zlib.so’, ‘zlib.dylib’, ‘zlib.dll’. In the case of a name (without directory part) the search path for shared objects is implemenation dependent.

For Unix We can find documentation about the search path of shared objects in the manual page of dlopen().

— Function: shared-object? obj

Return true if obj is a shared object.

— Function: lookup-shared-object shared foreign-symbol

Search the shared object referenced by shared for an exported symbol matching foreign-symbol, which must be a Scheme symbol or string, and return its address as a pointer object. Pointer objects.

If the foreign symbol is not found: raise a continuable exception with raise-shared-object-lookup-error. Shared objects lookup error.

— Constant: libc-shared-object

Bound to the shared object value identifying the C language runtime library. Notice that we can configure this value at installation time, with the --with-libc-shared-object option to the configure script.

— Syntax: define-shared-object <name> <library-name>

Expand to: 

          (define <name>
            (open-shared-object <library-name>))
