24.1.2 Defining labels

A label allows us to take an identifier (bound to whatever value) and define new identifiers representing virtual fields, methods, a setter and a getter, a parent label or class. Labels can inherit from a class, but not vice versa. Labels can be used by with-class and all its derived syntaxes; labels cannot be used to dispatch generic functions defined by the (nausicaa language generics) library.

— Macro: define-label name clause ...
— Macro: define-label (name predicate) clause ...

Define a new label type. Labels are somewhat like classes, but with no concrete fields, no underlying record type and no constructor protocols, no virtual methods; labels cannot be instantiated.

The following clauses are available in the body of a label definition:

inherit
Select a parent label or class; it has the same syntax and semantics of the corresponding clause for classes. The default is to inherit everything from the parent but concrete fields; this is fine when the parent is another label, but when the parent is a class we should use the concrete-fields option to enable bindings for the parent's concrete fields.

If no inherit clause is present: by default the label inherits from <top>, which is a class.

predicate
Select a predicate function to be used to identify values with the is-a? macro; it has the same syntax and semantics of the corresponding clause for classes. If a predicate is not selected: it defaults to a function which always returns #t.
virtual-fields
method
method-syntax
methods
setter
getter
bindings
They have the same syntax and semantics of the corresponding clauses for classes.
custom-maker
Define a maker which can be used to build whatever value the label provides an API to. The maker definition must be external to the define-label body.
shadows
Available only for labels. Allows a label to shadow another type.

The following is an example of library in which a label is used to handle lists as tuples of values: 

     #!r6rs
     (library (the-tuple)
       (export <tuple> a: b: c:)
       (import (nausicaa))
     
       (define-auxiliary-syntaxes
         a: b: c:)
     
       (define (tuple-predicate obj)
         (and (not (null? obj))
              (list? obj)
              (= 3 (length obj))))
     
       (define-maker make-tuple
         list
         ((a: #f)
          (b: #f)
          (c: #f)))
     
       (define-label (<tuple> tuple?)
         (predicate tuple-predicate)
         (virtual-fields (immutable a car)
                         (immutable b cadr)
                         (immutable c caddr))
         (custom-maker make-tuple))
       )

it can be used as follows: 

     #!r6rs
     (import (nausicaa) (the-tuple))
     
     (let (((o <tuple>) (make <tuple>
                          (a: 1)
                          (b: 2)
                          (c: 3))))
     
       (is-a? o <tuple>)     ⇒ #t
       (tuple? o)            ⇒ #t
       (list o.a o.b o.c)    ⇒ (1 2 3)
       )

The following meaningless example shows how labels and generic functions from (nausicaa language generics) can be used to define “interfaces” and classes implementing them: 

     ;; the interface <iface> has two methods
     
     (define-generic iface-one (o))
     (define-generic iface-two (o))
     
     (define-label <iface>
       (methods (one iface-one)
                (two iface-two)))
     
     ;; the classes <alpha> and <beta> implement <iface>
     
     (define-class <alpha>
       (fields a))
     
     (define-class <beta>
       (fields b))
     
     (define-method (iface-one (o <alpha>)) '(one . alpha))
     (define-method (iface-two (o <alpha>)) '(two . alpha))
     (define-method (iface-one (o <beta>))  '(one . beta))
     (define-method (iface-two (o <beta>))  '(two . beta))
     
     ;; tests
     
     (let (((p <iface>) (make <alpha> 1))
           ((q <iface>) (make <beta>  2)))
       (check (p.one) ⇒ '(one . alpha))
       (check (p.two) ⇒ '(two . alpha))
       (check (q.one) ⇒ '(one . beta))
       (check (q.two) ⇒ '(two . beta))
       )
