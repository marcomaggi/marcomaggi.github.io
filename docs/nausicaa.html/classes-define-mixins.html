24.1.13 Composing with mixins

Mixins are a way to add definition clauses to classes and labels. Let's consider this situation: 

     #!r6rs
     (import (nausicaa))
     
     (define-class <alpha>
       (field a))
     
     (define-class <beta>
       (field b))
     
     (define-class <delta>
       (inherit <alpha>)
       (field v)
       (method (doit (o <delta>))
         (+ 1 o.v)))
     
     (define-class <gamma>
       (inherit <beta>)
       (field v)
       (method (doit (o <gamma>))
         (+ 1 o.v)))

the definitions of <delta> and <gamma> share some clauses; both of them already have a parent class and multiple inheritance is not supported by Nausicaa/Scheme. Is it possible to write the shared clauses only once and attach them to the class definitions? Yes, with mixins.

The example above is equivalent to the following: 

     #!r6rs
     (import (nausicaa))
     
     (define-class <alpha>
       (field a))
     
     (define-class <beta>
       (field b))
     
     (define-mixin <stuff>
       (field v)
       (method (doit (o <stuff>))
         (+ 1 o.v)))
     
     (define-class <delta>
       (inherit <alpha>)
       (mixins <stuff>))
     
     (define-class <gamma>
       (inherit <beta>)
       (mixins <stuff>))

define-mixin associates a set of clauses to the identifier <stuff>; when the mixins clause is used in the body of a class definition:

  1. The clauses associated to the selected identifier are retrieved as syntax object.
  2. All the instances of the identifier <stuff> are substituted with the identifier of the enclosing class or label, <delta> and <gamma> in the example.
  3. The resulting clauses are added to the enclosing definition.

classes mixins for further examples. We can use the mixins clause in the definition body of classes, labels and mixins.

— Auxiliary Syntax: mixins ?spec ...

Compose the enclosing class, label or mixin definition with the given list of mixins. Each ?spec element must have one of the forms: 

          ?mixin-name
          (?mixin-name (?from ?to) ...)

in which ?mixin-name is the identifier referencing the mixin, ?from and ?to are identifiers specifying the specialisation for this mixin.

To specialise a mixin's clauses for the receiving definition means the following:

  1. All the occurrences of ?mixin-name are replaced with the receiving definition's class, label or mixin name.
  2. All the occurrences of the ?from identifiers are replaced with the ?to identifiers.

It is a syntax violation if a mixin identifier name is not already associated to a set of clauses at the time the receiving definition is expanded; it is a syntax violation if the same ?mixin-name is used multiple times in the mixins clauses of a definition.

The mixins clause can be used any number of times, each of them can list any number of mixin identifiers; the following clauses: 

     (mixins <a> <b>)
     (mixins <c> <d>)
     (mixins <e> <f>)

are equivalent to the single clause: 

     (mixins <a> <b> <c> <d> <e> <f>)

notice how the order is preserved from left to right and top to bottom.

Mixin clauses are added to the receiving definition in a deterministic order: first the original clauses, then the mixin clauses in the same order in which the mixin identifiers appear in the mixins clauses. For example the following definitions: 

     #!r6rs
     (import (nausicaa))
     
     (define-mixin <alpha>
       (fields second))
     
     (define-mixin <beta>
       (fields third))
     
     (define-mixin <gamma>
       (fields fourth))
     
     (define-class <red>
       (fields first)
       (mixins <alpha> <beta> <gamma>)
       (protocol
         (lambda (make-top)
           (lambda (first second third fourth)
             ((make-top) first second third fourth)))))

are equivalent to the following single definition: 

     #!r6rs
     (import (nausicaa))
     
     (define-class <red>
       (fields first second third fourth)
       (mixins <alpha> <beta> <gamma>)
       (protocol
         (lambda (make-top)
           (lambda (first second third fourth)
             ((make-top) first second third fourth)))))

notice how the order of the fields clauses matches the order of the arguments to the constructor function.

The capability of substituting arbitrary identifiers in the mixin clauses is a powerful tool of specialisation; for example, we could substitute identifier placeholders in the mixin clauses with class and namespace names for the receiving definitions (see namespaces): 

     #!r6rs
     (import (nausicaa))
     
     (define-auxiliary-syntaxes
       <sequence> <<for-sequence>>)
     
     (define-namespace <<for-string>>
       (functions (ref  string-ref)
                  (set! string-set!)))
     
     (define-namespace <<for-vector>>
       (functions (ref  vector-ref)
                  (set! vector-set!)))
     
     (define-mixin <the-sequence>
       (fields (subject <sequence>)
       (method (get-first (o <the-sequence>))
         (with-namespace ((N <<for-sequence>>))
           (N.ref o.subject 0))))
     
     (define-class <the-string>
       (mixins (<the-sequence>
                 (<sequence>       <string>)
                 (<<for-sequence>> <<for-string>>))))
     
     (define-class <the-vector>
       (mixins (<the-sequence>
                 (<sequence>       <vector>)
                 (<<for-sequence>> <<for-vector>>))))
     
     (let (((o <the-string>) (make* <the-string>
                               "ciao")))
       o.subject.length      ⇒ 4
       (o.get-first))        ⇒ #\c
     
     (let (((o <the-vector>) (make* <the-vector>
                               (vector 'c 'i 'a 'o))))
       o.subject.length      ⇒ 4
       (o.get-first))        ⇒ c
