1 Overview

Nausicaa/POSIX is a distribution of libraries for R6RS Scheme exposing the POSIX interface to the Scheme language level; some GNU C Library extensions and Linux extensions are also exposed.

The libraries are built on top of the Nausicaa/Scheme libraries, making use of their foreign-functions interface; some libraries depend upon the C language library ‘libnausicaa-posix’, which is built and installed by this package.

  • overview libraries: Libraries organisation.
  • overview interface: Interface organisation.

Next: overview interface, Up: overview

1.1 Libraries organisation

The distribution attempts to split functions compliant to the POSIX standard from functions available only through extensions; the standard C library functions are classified as extensions. This splitting is meant to make it easier to port the distribution on non–GNU systems; it is an ongoing process (the manual pages on GNU+Linux systems should contain notes about POSIX standard compliance).

Unless otherwise specified, libraries are split in three sets:

(foreign posix ---)
(foreign glibc ---)
(foreign linux ---)
The high–level interface: Slower but user friendly. We should really import these libraries by prefixing the identifiers; for example: 
          (import (rnrs)
            (prefix (foreign posix fd)   posix:)
            (prefix (foreign glibc file) glibc:))

(foreign posix --- primitives)
(foreign glibc --- primitives)
(foreign linux --- primitives)
The middle–level interface; does marshaling of arguments and return values. In this documentation, all the bindings from these libraries are prefixed with ‘primitive:’, but they are exported without this prefix.
(foreign posix --- platform)
(foreign glibc --- platform)
(foreign linux --- platform)
The low–level interface: Faster but raw. In this documentation, all the bindings from these libraries are prefixed with ‘platform:’, but they are exported without this prefix.

Additionally, the following libraries are installed:

(foreign posix sizeof)
Exports bindings for constant values (like ‘O_CREAT’) and data structure accessors and mutators.

Let's take as reference the file set of features; when the documentation states that a set of bindings is exported by the POSIX file libraries, it means that the bindings are exported by the following libraries: 

     (foreign posix file)
     (foreign posix file primitives)
     (foreign posix file platform)

when the documentation states that a set of bindings is exported by the Glibc file libraries, it means that the bindings are exported by the following libraries: 

     (foreign glibc file)
     (foreign glibc file primitives)
     (foreign glibc file platform)

when the documentation states that a set of bindings is exported by the Linux file libraries, it means that the bindings are exported by the following libraries: 

     (foreign linux file)
     (foreign linux file primitives)
     (foreign linux file platform)

When the documentation states that a set of bindings is exported by the sizeof libraries, it means that the bindings are exported by the (foreign posix sizeof) library.

1.2 Interface organisation

For all the functions only a brief description is given: We have to use the foreign function's own documentation to have all the details. Unless otherwise specified, the interface adheres to the following pattern for function x:

— Function: x arg ...
— Parameter: x-function
— Function: primitive:x arg ...
— Function: platform:x c-arg ...

This is an interface for the x() foreign function.

x and primitive:x do something with the arguments, which are regular Scheme values. They take the same arguments and return the same return values. If successful they return values representing the result; in case of error they raise an ‘&errno’ condition. C language system errors.

x does the job by applying to the arguments the current value of the x-function parameter, which must be a function; the parameter is initialised to primitive:x.

platform:x is the direct wrapper for the foreign function. It accepts C language level arguments (like C strings), and returns 2 values: the return value from the foreign function call and the value of ‘errno’ right after the call.

Notice that, for almost all the functions, the value of ‘errno’ is meaningful only if the return value from the foreign function call indicates an error; ‘errno’ may be non–zero even if the function call was successful.

For the meaning of the C level arguments we have to read the foreign function's original documentation.

This interface gives us 3 choices:

  1. To call directly the foreign function using the platform:x wrapper; in this case we are responsible to convert the Scheme arguments to the appropriate C language level values (beware to non–garbage collected values!).
  2. To call the Scheme adapter primitive:x; this gives us: automatic conversion of arguments, exceptions in case of errors, automatic retrying in case the foreign function call is interrupted by an ‘EINTR’ signal.
  3. To call the high–level, parameterised, x interface; this gives us everything primitive:x does and additionally allows us to replace the primitive function with a version that returns controlled values, or raises an error, and/or logs the operation. This feature can be used to test Scheme code execution paths that handle exceptions.

Example of usage of failing primitive fork function: 

     (define (fake-fork)
       (raise-errno-error 'fork EINVAL))
     
     (parametrise ((fork-function fake-fork))
       ---
       (fork)
       ---)

3.3.3 Reading a directory stream

The following functions manage reading the special stream objects used to access directory contents. Reading and closing a directory stream.

As a usage example for the low level functions, let's see how directory-entries is implemented: 

     (define (%directory-entries dir)
       (let loop ((entry  (readdir_r dir))
                  (layout '()))
         (if (pointer-null? entry)
             layout
           (loop (readdir_r dir)
                 (cons (dirent-name->string entry) layout)))))
     
     (define (directory-entries pathname)
       (with-compensations
         (%directory-entries (opendir/compensated pathname))))

to read a directory and sort the entries: 

     (list-sort string<? (directory-entries "/home/marco"))

to read a directory using the file descriptor and the GNU flag O_NOATIME: 

     (with-compensations
       (letrec ((fd (compensate
                        (open the-root
                              (if (number? O_NOATIME)
                                  (bitwise-ior O_RDONLY
                                               O_NOATIME)
                                O_RDONLY)
                              0)
                      (with
                       (close fd)))))
         (directory-entries/fd fd)))

3.3.5 Scanning a directory

The following functions implement the scan operation of directory entries. Scanning the Content of a Directory. All the following bindings are exported by the Glibc file libraries.

Usage example: Sort the entries in increasing lexicographic order, excluding the dots: 

     (scandir "/some/path/name"
              (make-scandir-selector-callback
                 (lambda (struct-dirent)
                   (let ((name (dirent-name->string struct-dirent)))
                     (not (or (string=? "."  name)
                              (string=? ".." name))))))
              platform:alphasort)

and if we want to sort the entries in decreasing lexicographic order: 

     (scandir "/some/path/name"
              (make-scandir-selector-callback
               (lambda (struct-dirent)
                 #t))
              (make-scandir-compare-callback
               (lambda (a b)
                 (let ((a (dirent-name->string a))
                       (b (dirent-name->string b)))
                   (cond ((string<? a b) +1)
                         ((string>? a b) -1)
                         (else            0))))))
— Function: scandir pathname selector compare
— Parameter: scandir-function
— Function: primitive:scandir pathname selector compare
— Function: platform:scandir pathname namelist selector compare

Scan the directory selected by pathname, which must be a string or symbol, and return a list of Scheme strings representing the names of the entries in the directory.

selector and compare must be raw pointers to C language functions, either native or built with make-c-callback. Calling Scheme functions.

The function referenced by selector takes a pointer to struct dirent as argument and must return: ‘1’ if the entry has to be included in the output, ‘0’ if it has to be excluded.

The function referenced by compare takes two pointers of type struct dirent ** as arguments and must return a ternary comparison result (as strcmp() does).

— Function: make-scandir-selector-callback scheme-function
— Function: make-scandir-compare-callback scheme-function
— Function: primitive:make-scandir-selector-callback scheme-function
— Function: primitive:make-scandir-compare-callback scheme-function

Given a Scheme function, return a callback pointer to be used as selector or compare argument to scandir. They are implemented as follows: 

          (define (make-scandir-selector-callback scheme-function)
            (make-c-callback int
              (lambda (struct-dirent)
                (if (scheme-function struct-dirent) 1 0))
              (void*)))
          
          (define (make-scandir-compare-callback scheme-function)
            (make-c-callback int scheme-function (void* void*)))
— Function Pointer: platform:alphasort
— Function Pointer: platform:versionsort

Raw pointers to comparison functions exported by the GNU C Library; they are meant to be used as compare arguments for the scandir function.

alphasort will sort the strings in increasing lexicographic order.

Everyone should mind his own buffers

Unfortunately there are Scheme implementations that insist in implementing buffering for custom ports. One can say that they are called “custom” to let the customers (that is us) decide how to handle the underlying device, buffering included. This is not the opinion of some Scheme implementers.

So with some implementations we have to turn off buffering and turn on non-blocking mode for the device.

Do it yourself

If we feel the need to implement our own wrappers, we may find useful to know that a custom read function can be implemented like this: 

     (define (custom-binary-read fd bv start count)
       (with-compensations
         (let* ((p   (malloc-block/c count))
                (len (read fd p count)))
           (do ((i 0 (+ 1 i)))
               ((= i len)
                len)
             (bytevector-u8-set! bv (+ start i)
                                 (peek-unsigned-char p i))))))
     
     (define (custom-textual-read fd str start count)
       (with-compensations
         (let* ((p   (malloc-block/c count))
                (len (read fd p count)))
           (do ((i 0 (+ 1 i)))
               ((= i len)
                len)
             (string-set! str (+ start i)
                          (integer->char (peek-unsigned-char p i)))))))

and a custom write function can be implemented like this: 

     (define (custom-binary-write fd bv start count)
       (with-compensations
         (let ((p (malloc-block/c count)))
           (do ((i 0 (+ 1 i)))
               ((= i count)
                (write fd p count))
             (poke-char! p i (bytevector-u8-ref bv (+ start i)))))))
     
     (define (custom-textual-write fd str start count)
       (with-compensations
         (let ((p (malloc-block/c count)))
           (do ((i 0 (+ 1 i)))
               ((= i count)
                (write fd p count))
             (poke-char! p i
                (char->integer (string-ref str (+ start i))))))))

6.11 Reading lines

The following functions manage reading lines from streams. Line–Oriented Input.

— Function: getline stream
— Parameter: getline-function
— Function: primitive:getline stream
— Function: platform:getline pointer-pointer pointer-count stream

Read characters from stream until a newline is found, then returns a string holding those characters and a terminating newline.

If end–of–file is encountered before the newline, but some characters where read: The acquired characters are stored in a string without terminating newline, and the string is returned.

If end–of–file is encountered before the newline, and no characters where read: An empty string is returned.

If an error occurs while reading: An exception is raised.

— Function: getdelim stream delimiter
— Parameter: getdelim-function
— Function: primitive:getdelim stream delimiter
— Function: platform:getdelim pointer-pointer pointer-count delimiter stream

Like getline but reads characters until delimiter is found. delimiter must be a Scheme character with an integer representation in the range [0, 255].

As a usage example, the following code reads one line at a time from a file and accumulates them in the lines variables: 

     (import (rnrs)
       (receive)
       (compensations)
       (foreign memory)
       (foreign cstrings)
       (foreign errno)
       (foreign ffi peekers-and-pokers)
       (foreign glibc streams)
       (prefix (foreign glibc streams platform) platform:))
     
     (let* ((pathname the-pathname))
       (with-compensations
         (letrec* ((S           (compensate
                                    (fopen pathname "w+")
                                  (with
                                   (fclose S))))
                   (*pointer    (malloc-small/c))
                   (*count      (malloc-small/c))
                   (getp        (lambda ()
                                  (pointer-ref-c-pointer *pointer 0)))
                   (free        (lambda ()
                                  (let ((p (getp)))
                                    (unless (pointer-null? p)
                                      (primitive-free p)))))
                   (lines       '()))
           (let loop ()
             (receive (result errno)
                 (platform:getline *pointer *count S)
               (cond ((ferror S)
                      (free)
                      (raise-errno-error 'reading-line errno S))
     ;;; here EOF was found, return the result
                     ((= -1 result)
                      (free)
                      (reverse lines))
     ;;; accumulate the result
                     (else
                      (set! lines
                            (cons (cstring->string (getp) result)
                                  lines))
                      (loop))))))))

7.1 Introduction

Time handling is a complex topic, so we really want to read some discussion on it; for example the introduction in the GNU C Library documentation. Time Basics.

About the time_t type

The time_t type is used by several functions, but it is not fully specified2; it represents the epoch: The number of seconds elapsed since 00:00:00 on January 1, 1970, Coordinated Universal Time.

To solve the representation problem, time_t values are casted to native double values and then converted to Scheme inexact real numbers, whatever they are for the underlying Scheme representation.
